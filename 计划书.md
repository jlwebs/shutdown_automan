# 远程重启服务程序 项目计划书

## 1. 项目背景与目标

本项目的目标是开发一款轻量级的 Windows 系统托盘程序，用于远程触发系统重启，并支持自定义重启前的预处理逻辑。程序需提供简单易用的图形化配置界面，允许用户设置 HTTP 监听端口、需要终止的进程列表（每个进程可指定终止后的等待时间），并支持自动监控进程状态，在检测到指定进程缺失时自动执行重启流程。

该程序主要面向需要在无人值守或远程环境下，确保关键进程始终运行或在必要时重启系统的场景，例如工控机、自助终端、服务器等。

## 2. 功能需求

### 2.1 核心功能
- **HTTP 服务端**：在指定端口启动 HTTP 服务，提供 `/restart` 端点（仅接受 POST 请求），收到请求后执行重启逻辑。
- **重启逻辑**：按配置顺序执行以下操作：
  1. 依次终止每个指定进程（强制终止，如 `taskkill /f /im`）。
  2. 每个进程终止后，等待该进程配置的延迟时间（单位：秒）。
  3. 所有进程处理完毕后，执行系统重启命令（`shutdown /r /t 0`）。
- **进程列表配置**：支持以列表形式添加、删除、编辑进程项，每项包含：
  - 进程名（如 `notepad.exe`）
  - 终止后延迟秒数（整数，范围 0~3600）

### 2.2 自动监控功能
- **监控开关**：可启用/禁用自动监控。
- **监控间隔**：可配置检查周期（单位：秒，建议 1~3600）。
- **监控逻辑**：按配置的进程列表顺序，检查每个进程是否存在（通过 `tasklist` 或类似工具）。若任一进程缺失，则立即触发完整的重启逻辑（按配置顺序执行终止+延迟+重启），不再继续检查后续进程。

### 2.3 托盘界面
- **系统托盘图标**：程序启动后隐藏主窗口，仅显示托盘图标。
- **右键菜单**：
  - **设置**：打开配置对话框。
  - **启动服务**：启动 HTTP 服务。
  - **停止服务**：停止 HTTP 服务。
  - **帮助**：显示使用说明。
  - **退出**：停止服务并退出程序。
- **左键双击**：打开设置对话框（快捷操作）。
- **状态反馈**：服务启动/停止、自动触发重启等事件通过托盘气泡提示；错误信息通过消息框显示。

### 2.4 配置持久化
- 所有配置（端口、进程列表、监控开关、监控间隔）保存到本地 JSON 文件（如 `config.json`），程序启动时自动加载，修改后即时保存。
- 支持配置的自动迁移：若旧版配置文件使用逗号分隔的进程名字符串，应能转换为新版进程列表格式（延迟默认为 0）。

### 2.5 其他
- **帮助说明**：右键菜单“帮助”应弹出简洁的使用说明，包括基本操作、注意事项（如管理员权限要求）。
- **管理员权限**：程序需要以管理员身份运行，以执行进程终止和系统重启。需通过 manifest 或运行时检测提示用户。

## 3. 非功能需求

### 3.1 性能与资源占用
- **内存占用**：程序应尽量小巧，内存占用控制在 10MB 以内（典型值）。
- **CPU 使用**：空闲时无 CPU 消耗；监控定时器触发时短暂检查进程，不应影响系统性能。

### 3.2 平台兼容性
- 仅支持 **Windows 7/8/10/11**（64 位/32 位），依赖 Win32 API。
- 无需跨平台支持。

### 3.3 可靠性
- 服务启动/停止应稳定，端口冲突时应给出明确提示。
- 自动监控不应因单个进程检查失败而中断，应继续后续检查。
- 重启命令发出后，程序应能正常退出或等待系统关闭。

### 3.4 安全性
- HTTP 服务默认监听本地地址（127.0.0.1），避免暴露到公网。如需远程访问，建议通过防火墙或 VPN 控制。
- 可考虑添加简单的 API 密钥验证（可选扩展）。

### 3.5 易用性
- 配置界面直观，进程列表支持动态增删改。
- 操作有明确反馈，避免用户困惑。

## 4. 技术选型建议

### 4.1 编程语言：Go
- **理由**：
  - 编译为单一可执行文件，无外部依赖，分发简单。
  - 标准库强大，尤其是 `net/http` 和 `os/exec`，便于实现 HTTP 服务和调用系统命令。
  - 并发支持良好，便于处理监控定时器和 HTTP 服务。
  - 内存占用低，符合轻量级要求。
  - 社区有成熟的 GUI 库（如 Walk）可满足原生 Windows 界面需求。

### 4.2 GUI 框架：Walk（仅 Windows）
- **理由**：
  - 直接封装 Win32 API，生成原生窗口和控件，外观与系统一致。
  - 内存占用极小，无额外渲染引擎。
  - 提供托盘图标、菜单、对话框、表格视图等所需组件。
  - 开发周期短，代码量少，符合快速开发需求。
  - 相比 Fyne（自绘）或 Webview（需前端），Walk 更符合“原生”和“内存小”的要求。

### 4.3 配置格式：JSON
- **理由**：
  - 轻量、易读写，Go 标准库 `encoding/json` 支持完善。
  - 人类可读，便于手动修改或调试。

### 4.4 进程管理：`os/exec` 调用系统命令
- **终止进程**：`taskkill /f /im <进程名>` 简单可靠。
- **检查进程存在**：`tasklist /FI "IMAGENAME eq <进程名>"` 并解析输出。
- **重启系统**：`shutdown /r /t 0`。
- **理由**：无需额外库，直接利用 Windows 内置工具，稳定高效。

### 4.5 定时监控：`time.Ticker`
- **理由**：Go 原生定时器，轻量且准确，适合周期性检查。

## 5. 系统设计概要（可选）

### 5.1 模块划分
- **主模块**：程序入口，初始化托盘图标、加载配置、启动监控。
- **配置管理模块**：负责配置的加载、保存、迁移，以及设置对话框的数据绑定。
- **HTTP 服务模块**：管理服务的启动/停止，定义 `/restart` 处理函数。
- **重启执行模块**：接收进程列表，按顺序执行终止+延迟+重启，支持异步调用（避免阻塞 HTTP 响应或监控线程）。
- **监控模块**：基于配置的开关和间隔，定期检查进程，触发重启逻辑。
- **UI 模块**：托盘图标、菜单、设置对话框（使用 Walk 构建）。

### 5.2 数据流
- 用户通过设置对话框修改配置 → 配置保存到 JSON 文件 → 监控模块重启（若启用）以应用新间隔/列表。
- HTTP 请求 `/restart` → 调用重启执行模块（异步）→ 返回“已执行”响应。
- 监控定时器触发 → 检查进程 → 若缺失则调用重启执行模块。

### 5.3 线程安全
- 使用互斥锁保护共享状态（如服务运行标志、监控协程生命周期）。
- UI 操作需通过 `walk.App().Sync()` 切换到主 UI 线程执行，避免跨线程冲突。

## 6. 开发与部署建议

### 6.1 开发环境
- Go 1.16+（推荐 1.20+）
- 依赖库：`github.com/lxn/walk` 及相关（`github.com/lxn/win`）
- 编辑器：任意支持 Go 的 IDE（如 VS Code、GoLand）

### 6.2 编译与打包
- 使用 `go build -ldflags="-H=windowsgui"` 生成无控制台窗口的 GUI 程序。
- 需准备 `app.manifest` 文件，设置 `requireAdministrator` 权限，并与可执行文件置于同一目录（或使用 `rsrc` 工具嵌入）。
- 最终发布包应包含：`远程重启服务.exe` + `app.manifest`（可选，若未嵌入）。配置文件由程序自动生成。

### 6.3 测试要点
- 功能测试：验证 HTTP 触发、监控触发、进程终止、延迟等待、重启命令。
- 异常测试：端口占用、进程不存在、taskkill 失败、shutdown 被阻止等情况下的错误提示。
- 权限测试：以普通用户运行时应有明确提示或以管理员身份自动重启（通过 manifest 实现 UAC 弹窗）。
- 配置迁移测试：旧版配置文件能否正确转换为新版格式。

### 6.4 扩展可能性
- **API 认证**：增加请求头验证（如 X-API-Key），防止未授权调用。
- **日志记录**：将操作日志写入文件，便于追溯。
- **多语言支持**：可扩展界面语言包（Walk 支持国际化）。
- **更精细的进程检测**：如需检测“无响应”状态，可借助 `enumprocesses` 或 `GetProcessTimes` 等 API，但会增加复杂度。
